# Reliability phase ordering: structured errors first, then hot-plug, then auto-reconnect

The three reliability primitives for v1.0 are structured errors + observability, hot-plug detection, and auto-reconnect. They are not independent and must be built in a specific order. **Structured errors come first** because both hot-plug and auto-reconnect depend on the project being able to *categorise* a failure as transient (network blip, device briefly unavailable) vs. permanent (device physically removed, server shut down) vs. fatal (corrupt state, protocol violation). That categorisation is a structured-error design decision, and retrofitting it onto an `eprintln!`-based codebase is more expensive than designing it in. **Hot-plug comes second** because it requires the runtime to *observe* device-attach and device-detach events from the OS (libusb hotplug callbacks on Linux, `RegisterDeviceNotification` on Windows, `UsbManager` intents on Android) and surface them as structured events, not crashes. The platform-specific code for that is non-trivial. **Auto-reconnect comes last** because it is the simplest of the three once the other two exist: given structured errors (know what's transient), given device-presence events (know when to stop retrying), the network-level retry / backoff / resume flow is straightforward.

## Deferred

Session persistence (resuming an active import after a server crash, preserving the device-import state) is explicitly *not* in v1.0. It is a stronger guarantee than auto-reconnect and requires stable serialised state on both sides.
